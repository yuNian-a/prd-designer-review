# Output Template

需求预审的输出结构模板，包含完整的输出格式、风格要求与执行限制。

执行时必须严格按照以下结构输出，完整填充每个部分，不得跳过，不得改变标题层级或顺序。

---

# 需求预审结果

## 1. 需求概述
用简洁语言概括：

- 这是一个什么需求
- 目标用户是谁
- 用户要完成的核心任务是什么
- 涉及哪些关键页面 / 模块
- 当前需求是否可大致还原主流程

## 2. 主流程还原
按步骤列出主链路：

1. 用户从哪里进入
2. 用户做什么
3. 系统如何反馈
4. 用户如何完成目标
5. 存在哪些关键分支 / 状态 / 例外情况

若流程无法完整还原，要直接指出缺失点。

## 3. 用户视角问题
按以下格式逐条输出：

### [优先级（高/中/低）+ 问题标题]
- **所在流程 / 步骤**：
- **判断依据**：（已知 / 推断 / 缺失，说明来源）
- **问题描述**：
- **为什么这是问题**：
- **对用户的影响**：
- **行业 / 常见模式参考判断**：
- **修正建议**：

## 4. 设计师视角问题
按以下格式逐条输出：

### [优先级（高/中/低）+ 问题标题]
- **所在模块 / 页面 / 规则点**：
- **判断依据**：（已知 / 推断 / 缺失，说明来源）
- **问题描述**：
- **矛盾 / 缺失点**：
- **为什么会影响设计落地**：
- **建议补充或修正内容**：

## 5. 需重点带入线下评审的问题

1. 问题
2. 为什么必须确认
3. 建议向产品经理追问什么

## 6. 是否建议进入设计阶段
给出明确判断：

- **可直接进入设计**
- **可部分进入设计，但需先确认若干高风险问题**
- **暂不建议进入设计，需先补齐关键信息**

并说明原因。

---

## 输出风格要求

- 保持专业、克制、清晰
- 不使用空泛评价，如"整体较合理""建议进一步优化"
- 不只描述现象，必须说清原因和影响
- 不要用产品经理口吻为需求辩护
- 不要为了显得全面而制造低价值问题
- 问题数量以有效为准，优先指出真正影响设计判断和用户体验的问题
- 当信息不足时，明确写出"当前文档未提供，无法判断"

---

## 执行限制

执行时必须避免以下行为：

- 不得只复述 PRD 内容
- 不得把需求摘要当作评审结果
- 不得只从功能是否存在出发，而忽略用户任务是否成立
- 不得默认"有页面描述 = 可以设计"
- 不得把纯技术实现问题作为主要结论
- 不得臆造需求中未出现的规则
- 不得用模糊表述代替判断，例如"可能需要考虑""似乎存在问题"
