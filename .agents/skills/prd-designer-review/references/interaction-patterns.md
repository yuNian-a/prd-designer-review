# Interaction Patterns

通用交互模式参考，用于在需求预审时判断需求的交互设计是否符合行业惯例，识别与常见模式的偏差。

---

## 一、表单

### 通用规范
- 必填字段需有明确标识（通常为红色星号 `*`）
- 校验时机：提交时全量校验 + 离焦时单字段校验（两者缺一易造成用户困惑）
- 错误提示需出现在对应字段下方，而非仅顶部 toast
- 提交成功后需有明确反馈（跳转 / 成功提示）

### 常见问题模式
- 需求只写了"填写表单"，但未定义哪些字段必填、校验规则是什么
- 编辑态与新建态复用同一表单，但规则不同，未分开说明
- 表单字段存在联动关系（如选择 A 后显示 B），但联动规则未定义

---

## 二、列表 / 表格

### 通用规范
- 默认排序规则需定义（时间倒序 / 正序 / 权重）
- 分页参数需明确（每页条数默认值、是否支持修改）
- 列表需定义空状态（无数据时的展示内容和引导）
- 批量操作需说明：哪些操作支持批量，批量操作的反馈如何处理

### 常见问题模式
- 筛选条件定义了，但默认值未说明
- 搜索范围未定义（搜索哪些字段，是否模糊匹配）
- 列表操作列中的按钮在不同状态下的显隐规则未说明

---

## 三、弹窗 / 抽屉

### 使用场景区分
| 组件 | 适用场景 |
|---|---|
| Modal 弹窗 | 二次确认、轻量表单（3 个字段以内）、简短信息展示 |
| Drawer 抽屉 | 详情预览、较多字段的表单填写、不需要离开当前上下文 |
| 新页面 | 流程复杂、步骤多、需要独立浏览器 Tab 操作的场景 |

### 常见问题模式
- 需求说"弹窗填写"，但字段超过 8 个，更适合用抽屉或新页面
- 弹窗关闭时，未定义数据是否保留（用户误关后是否丢失填写内容）
- 弹窗内的操作结果（保存/提交）如何刷新背后的列表页，未说明

---

## 四、状态与反馈

### Toast / 消息提示
- 成功操作 → 绿色 toast，自动消失（2~3s）
- 警告 / 提示 → 黄色，视情况可手动关闭
- 错误 → 红色，通常需要手动关闭或引导用户修正

### 按钮状态
| 状态 | 使用条件 |
|---|---|
| 默认可用 | 操作可执行 |
| 禁用（disabled） | 条件未满足，但需要告知用户原因（tooltip 或说明文案） |
| 加载中（loading） | 操作提交中，防止重复点击 |
| 隐藏 | 当前角色 / 状态下完全不应呈现该操作 |

### 常见问题模式
- 按钮在某些状态下说"置灰"，但没说明置灰的触发条件和是否有提示
- 操作成功后未定义页面如何刷新或跳转

---

## 五、导航与跳转

### 面包屑
- 超过 2 层的页面结构应提供面包屑
- 面包屑节点需可点击返回

### 返回行为
- 返回是否会丢失未保存数据？若会，需二次确认
- 多层级弹窗关闭后，返回哪一层？

### Tab 与多视图
- Tab 切换时，当前 Tab 的未保存数据如何处理？
- Tab 数量变化（动态 Tab）的规则是否说明？
