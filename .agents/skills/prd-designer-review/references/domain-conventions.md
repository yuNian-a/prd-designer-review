# Domain Conventions

特定行业的常见流程结构与设计惯例，用于在需求预审时判断当前方案是否偏离行业标准模式，以及偏离是否有足够的业务理由。

---

## 使用说明

本文件收录了常见行业场景的设计惯例。AI 在评审时，应对照当前需求所属行业，判断：
- 当前方案是否与行业惯例一致
- 若不一致，是否有明确的业务收益支撑这种差异
- 若没有足够理由，应标记为中或高风险问题

---

## 一、ToB 管理后台（通用）

### 列表页标准结构
- 顶部：筛选条件区 + 搜索框
- 右上角：主操作按钮（新建 / 导入）
- 表格：数据列表，末列为操作列
- 底部：分页器

### 详情页标准结构
- 顶部：面包屑 + 页面标题 + 操作按钮区
- 主体：分组展示的基本信息
- 关联数据：以 Tab 或分区形式展示

### 审批流标准心智
- 状态通常为：草稿 → 待审批 → 审批中 → 已通过 / 已拒绝
- 拒绝后支持修改重新提交
- 审批记录（时间轴）是高频需求，PRD 经常遗漏

### 权限管理标准结构
- 角色 → 权限集合的映射
- 页面级权限 vs 操作级权限需分开定义
- "可见但不可操作"需要有明确的视觉区分

## 二、RWA 官网类产品（Finloop RWA Official Website）

> 适用于科技感黑色主题的品牌官网、营销页面类需求评审。

### 页面结构标准

当前项目包含以下固定页面，新增页面需说明与现有路由的关系：

| 页面 | 路由 | 职责 |
|---|---|---|
| 首页 | `/` | 品牌入口，核心卖点 + CTA |
| 产品列表 | `/products` | 产品矩阵展示 |
| 产品详情 | `/products/:id` | 单个产品深度介绍 |
| 新闻列表 | `/news` | 内容运营入口 |
| 新闻详情 | `/news/:id` | 文章阅读页 |
| 关于我们 | `/about` | 品牌背书 |

### 页面区块构成惯例

每个页面由若干独立区块（Section）纵向叠加构成，每个区块职责单一：

- **Hero 区块**：Eyebrow 标签 + 大标题 + 副标题 + CTA 按钮组，位于页面最顶部
- **功能/特性区块**：以卡片网格或分栏排布，说明产品核心价值
- **数据/指标区块**：数字化展示品牌背书（资产规模、用户数等）
- **CTA 区块**：每个页面至少有一个引导联系 / 注册的区块，通常在页尾

若需求新增区块，必须说明：区块目标、位置、内容构成、CTA 指向。

### 多语言要求

**所有页面内容必须支持三语**（英文 / 简体中文 / 繁体中文）。  
若需求文档中的文案只提供了一种语言，需在评审问题中标注为缺失，要求补充另外两种语言版本。

- 不允许某个页面或区块只有中文无英文，或反之
- 动态内容（产品名、新闻标题）也需要多语言支持，若 PRD 未说明来源语言，应标注为缺失

### 视觉规范约束（影响设计可落地性的关键点）

以下是当前项目已确定的视觉规范，若需求描述的设计方案与之冲突，应标注为设计风险：

- **主题**：全站纯黑背景（`#000`），不接受白底或浅色区块方案（除非有明确业务理由）
- **大标题**：必须使用白色渐变文字效果（顶部白 → 底部灰），不接受实色标题
- **Eyebrow 标签**：小标题使用等宽字体（Roboto Mono）+ 全大写 + 字间距，风格固定
- **卡片**：使用玻璃拟态效果（半透明渐变背景 + 边框），不接受纯色卡片
- **按钮**：描边按钮（次要）和实心白色按钮（主要），两者风格固定，不接受其他样式变体

### 响应式要求

所有页面和区块必须支持移动端适配，断点为 768px 和 480px。  
若需求文档未提供移动端布局说明，应标注为缺失项，不能默认与桌面端相同。

### 联系表单惯例

当前项目统一使用 `ContactModal` 弹窗收集联系信息，不单独开发联系页面表单。  
若需求要求新增独立的表单页面，需说明与现有弹窗的区别和必要性。

## 三、AI Agent 对话平台（FAI — AI Agent Web）

> 适用于企业级 AI 助理 / Agent 对话平台类需求评审。

### 项目核心模块

| 模块 | 路由 | 职责 |
|---|---|---|
| 首页 | `home/` | 产品入口，引导用户开始对话 |
| AI 对话 | `ChatView.vue` | 核心对话界面，最高频使用路径 |
| Agent 市场 | `skillMarket/` | Agent / 技能选择与管理 |
| 个人助理 | `personalAssistant/` | 用户专属 Agent 配置 |
| 团队管理 | `team/` | 多人协作与权限管理 |
| 新闻资讯 | `news/` | 7x24 快讯 + 资讯列表 |
| 数据看板 | `dashboard/` | 使用数据统计 |
| 个人中心 | `profile/` | 账号设置 |

### 对话界面核心惯例

对话（ChatView）是本产品最核心的用户路径，评审对话相关需求时重点检查：

**流程完整性**
- 新建对话 → 选择 Agent → 输入内容 → 收到回复 → 对话持续 → 保存历史，每个节点都需要定义
- AI 回复过程中必须有流式输出状态（逐字显示），PRD 未说明流式体验则标注为缺失
- 对话失败（网络错误 / 超时 / 模型错误）必须有明确的错误提示 + 重试入口

**历史记录**
- 历史对话通过 `HistoryPanel`（侧边面板）和 `HistoryDrawer`（抽屉）两种形式呈现，需求需说明使用哪种以及触发方式
- 历史记录支持重命名（`EditTitleModal`）和删除（`DeleteConfirmModal`），若需求涉及这两个操作，交互规则必须明确
- 若需求要求新增历史记录的其他操作（收藏、分享、导出），需说明入口位置

**输入区域**
- 输入框组件为 `ChatInput.vue`，已支持：多行文本输入、发送按钮、回车发送
- 若需求要求输入区新增功能（附件上传、语音输入、@功能等），属于需要新开发的能力，需标注为设计风险

### Agent 市场惯例

- Agent 选择通过 `AgentSelector` 完成，列表项为 `AgentItem`
- 若需求描述了"Agent 推荐""Agent 搜索""Agent 分类筛选"等功能，需明确这些是新增能力还是已有功能扩展
- Agent 详情页（介绍、能力、使用示例）是否存在、入口在哪，需求必须明确

### 团队协作惯例

- 团队模块涉及多角色权限，需求必须说明：管理员 / 成员 / 访客各自能看到什么、能操作什么
- 共享对话与私人对话需有明确的视觉区分
- 团队成员管理（邀请、移除、角色变更）每个操作的权限范围必须定义清楚

### 布局结构约束

本项目采用**左侧边栏（`NewSidebar`）+ 顶部导航（`GlobalHeader`）+ 主内容区**的固定布局：
- 若需求要求改变整体布局结构，需说明充分理由，否则标注为高风险
- 弹窗统一使用 `DeleteConfirmModal`（删除确认）和 `EditTitleModal`（标题编辑）这两个现有组件，若需求需要新弹窗类型，需说明差异
- 首次登录引导通过 `FirstLoginDialog` 实现，若需求涉及新手引导流程，应复用此组件而非重新设计

### 新闻资讯模块惯例

- 分为 7x24 快讯（实时流）和资讯列表（文章）两种内容形态，需求应说明新内容属于哪种形态
- 若需求要求新增内容类型（视频、播客等），需明确与现有两种形态的差异

## 四、Finloop 官网（finloop-website-web）

> 适用于面向金融机构的企业品牌官网类需求评审。

### 项目定位与页面结构

面向金融机构的 B2B 企业官网，支持三语言、PC + 移动端双端：

| 页面 | 路由 | 主组件 |
|---|---|---|
| 首页 | `/` | `Index.vue` |
| 产品介绍 | `/products` | `Products.vue` |
| 解决方案 | `/solution` | `Solution.vue` |
| 全球布局 | `/global` | `Global.vue` |
| 关于我们 | `/us` | `Us.vue` |

> 路由带语言前缀时变为 `/zhCn/`、`/enUs/`，繁体中文无前缀（默认语言）。

### 多语言要求

**必须支持三语**：简体中文（`zhCn`）/ 繁体中文（默认）/ 英文（`enUs`）。

- 所有页面文案必须同步提供三种语言版本
- 若 PRD 只提供中文文案而未说明其他语言的处理方式，标注为缺失
- 部分图片有多语言版本（标题图、配图），需求需说明图片是否需要按语言切换

### 布局结构约束

- 顶部导航（`HeaderNav`）和底部（`Footer`）已由全局 Layout 引入，**不得**在业务需求中要求移除或重新设计这两个区域
- 每个业务页面由若干独立区块纵向叠加，区块之间彼此独立
- 若需求要求改变整体布局（如取消顶导、侧边导航），需说明充分理由

### 响应式要求

本项目断点为 **1024px / 750px / 480px**（与其他项目不同）：
- PC 端：> 1024px
- 平板：≤ 1024px
- 移动端：≤ 750px

若需求只提供了 PC 端布局方案而未说明移动端适配，标注为缺失。  
移动端与 PC 端可能使用不同的 UI 组件（PC 用 Element UI，移动端用 Vant），若需求涉及复杂交互组件，需说明双端实现方式是否一致。

### 图片使用惯例

- 大图、背景图、产品图 → 通过 CDN 加载（`ImgCDN` 组件）
- 若需求新增图片内容，必须说明图片来源（由设计提供上传 CDN，还是使用已有 CDN 资源）
- 需要按语言显示不同图片的情况（如标题图），PRD 必须说明有哪几个语言版本

### 视觉风格约束

本项目以**浅色为主基调**（非全黑），但有深色区块：
- 主要内容区：白色/浅灰背景（`#fff` / `#f9fafb`）
- 深色区块（如 Hero 区、CTA 区）：`#00152A` 深蓝背景
- 品牌主色：`#1B83F8`（蓝色），与 RWA 官网的 `#3467FF` 不同，**两个项目不可混用**

## 五、TaaS 平台（Webportal — RWA 资产代币化管理平台）

> 适用于面向机构的 RWA 代币化管理平台类需求评审。

### 项目定位与用户角色

面向机构的 B2B 金融资产代币化管理平台，用户角色分三类：
- **管理员**：平台级管理权限
- **机构用户**（投资者 / 分销商）：资产查看、交易操作
- **员工**：内部操作人员

PRD 中每个页面、每个操作必须明确所属角色，不同机构类型（Institution Type）和用户类型（User Type）看到的菜单不同。若需求未说明权限范围，标注为高风险缺失。

### 主要业务模块

| 模块 | 路由前缀 | 核心职责 |
|---|---|---|
| 代币化 | `/tokenization` | 资产代币发行管理 |
| 项目代币 | `/projectToken` | 代币项目详情 |
| 合同模板 | `/contractTemplate` | 合同模板管理 |
| 交易记录 | `/tradeRecord` | 交易流水列表与详情 |
| 交易审核 | `/tradeAudit` | 审核列表与详情 |
| 代币价格 | `/tokenPriceManage` | 价格管理 |
| 持仓 | `/holdings` | 持仓查看 |
| 客户 | `/customer` | 客户列表与钱包地址 |
| 分销商 | `/distributor` | 分销商列表与钱包地址 |
| 账户 | `/account` | 账户信息与钱包 |
| 员工 / 权限 | `/employee` `/userManagement` `/roleManagement` | 员工与角色权限管理 |
| 登录 | `/` | 含初始密码设置、机构类型选择、找回密码 |

### 权限体系评审要点

本项目为**菜单级 + 按钮级**双层权限控制，通过 `authCode` 字段管理：
- PRD 中每个操作按钮必须标注 `authCode`，否则开发无法实现权限控制，标注为高风险缺失
- 同一页面在不同角色 / 机构类型下可能有完全不同的操作集合，需求需分角色说明
- "可见但不可操作"与"不可见"是两种不同权限状态，PRD 必须区分说明

### 多语言与多主题要求

- **三语**：简体中文（`zhCn`）/ 繁体中文（`zhTc`）/ 英文（`enUS`）
- **双主题**：亮色（light）/ 暗色（dark），通过 `data-theme` 切换
- 所有颜色必须使用 CSS 变量（见 `design-tokens.md`），**禁止硬编码色值**
- 若需求涉及新增颜色或视觉方案，需确认该色值在 light / dark 两个主题下均合理

### 金融场景核心约定

评审涉及金融数据展示的需求时，以下内容必须在 PRD 中明确，否则标注为缺失：

| 约定项 | 规则 | 未说明时的风险 |
|---|---|---|
| 涨跌颜色 | 红涨绿跌（中国市场惯例） | 若有海外用户，可能需要反转 |
| 金额精度 | 默认 2 位小数，份额类需单独约定 | 精度不一致导致计算显示问题 |
| 千分位 | 大额数值需展示千分位 | 可读性差 |
| 时间格式 | `YYYY-MM-DD HH:mm:ss`，时区默认 UTC+8 | 跨时区用户显示错误 |
| 币种 | 是否支持多币种，币种选择逻辑 | 多币种场景未处理导致显示异常 |
| 空值展示 | 数据为空时展示 `--` 还是 `0` 还是空 | 用户误解数据含义 |

### 交易 / 审批类流程评审要点

本平台大量涉及交易和审批流程，评审此类需求时重点检查：
- 交易发起 → 审核 → 完成 / 拒绝 的完整状态流转是否定义清楚
- 审核拒绝后，发起方如何得知、能否修改重新提交
- 交易记录的不可修改性（区块链特性），撤销场景需特别说明
- 钱包地址的展示与复制必须有明确交互规则（`FCopy` 组件）

## 六、Finloop Enterprise Web（EPS / MFO 金融资产管理平台）

> 适用于面向机构投资者 / 财富管理从业者的 B 端金融 SaaS 平台类需求评审。

### 项目结构与子工程

Monorepo 多入口结构，需求评审时需先确认需求属于哪个子工程：

| 子工程 | 面向用户 | 核心业务 |
|---|---|---|
| `eps` | 机构投资者 | 公募基金、私募基金、债券、票据、RWA、保险、账户管理 |
| `mfo` | 多家庭办公室 | 在 eps 基础上增加客户管理、渠道管理 |

同一需求若同时影响 eps 和 mfo，必须在 PRD 中分别说明两端的差异，不能用"同 eps"一笔带过。

### 主要业务模块

**EPS 端：**

| 模块 | 路径 | 说明 |
|---|---|---|
| 账户 | `account/` | 总览、资产明细、对账单 |
| 公募基金 | `publicFund/` | 产品列表、申购、赎回、定投、持仓 |
| 私募基金 | `privateFund/` | 含稳健型子模块 |
| 债券 | `bond/` | 产品、询价、订单 |
| 票据 | `note/` | 产品、询价、订单 |
| RWA | `rwa/` | 产品、订单 |
| 保险 | `insurance/` | 保险产品 |
| 首页 | `home/` | 用户首页 |
| 用户 | `user/` | 密码、默认机构设置 |
| 员工 / 权限 | `employee/` | 用户管理、角色权限管理 |

**MFO 端额外模块：**

| 模块 | 路径 | 说明 |
|---|---|---|
| 客户管理 | `customer/` | 客户资产管理 |
| 渠道管理 | `accountManage/` | 渠道、用户、角色管理 |
| MFO 首页 | `mfoHome/` | 专属首页（文章、Banner） |

### 权限体系评审要点

EPS 端存在员工权限体系（用户管理 + 角色管理），涉及权限的需求必须说明：
- 哪些功能需要权限门控
- 不同角色的操作差异（可见 vs 可操作）
- 权限不足时的页面降级方案（禁用、隐藏还是跳转）

### 金融场景核心约定

与 TaaS 平台共享以下约定，评审时同样适用：

| 约定项 | 规则 |
|---|---|
| 涨跌颜色 | 红涨绿跌，固定不可更改 |
| 金额精度 | 需在 PRD 中明确，不同产品类型精度不同 |
| 千分位 | 大额数值需展示 |
| 时间格式 | `YYYY-MM-DD HH:mm:ss`，时区明确 |
| 状态值 | 必须对应后端枚举，PRD 中的状态标签需标注枚举值 |
| 空值 | 数据缺失时展示规则需明确（`--` / `0` / 空） |

### 交易类流程评审要点

本平台涵盖多个金融产品的申购/赎回/询价流程，评审时检查：
- 申购 / 赎回 / 询价 每种操作的完整流程（发起 → 确认 → 提交 → 结果）是否定义
- 交易密码验证步骤（`TradePWDModel`）是否在需要的环节说明
- 订单状态流转（待确认 → 处理中 → 已完成 / 已撤销）是否完整定义
- 不同产品类型（公募 / 私募 / 债券）的申购规则是否有差异，差异是否说明

### 新增页面的约束

路由无自动生成，新增页面需手动注册，PRD 必须明确：
- 页面路由路径
- 路由参数（如 `id`、`type` 等动态参数）
- 从哪些入口跳转过来，跳转时携带哪些参数
- 页面在哪个子工程（eps / mfo）中存在

若以上任一项缺失，标注为设计推进的阻塞问题。

## 七、其他项目自定义惯例

> 在此填写其他项目特有的行业惯例或内部规范：
