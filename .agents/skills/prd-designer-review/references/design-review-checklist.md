# Design Review Checklist

设计审查补充维度与常见问题模式，用于在 SKILL.md 核心评审框架之外，进一步细化检查项。

---

## 一、易遗漏的流程断点

在评审需求时，以下场景是高频遗漏区，需重点关注：

### 创建 / 提交类流程
- [ ] 草稿是否保存？草稿的入口和状态如何体现？
- [ ] 提交成功后落地页是什么？
- [ ] 提交后是否有后续引导动作？
- [ ] 是否需要二次确认？什么情况下需要？

### 审批 / 多级流转类流程
- [ ] 各节点审批人是谁？谁来指定？
- [ ] 审批拒绝后，发起人如何得知？能否修改后重新提交？
- [ ] 审批撤回的条件是什么？撤回后状态如何变化？
- [ ] 超时未审批怎么处理？

### 编辑 / 修改类流程
- [ ] 哪些状态下允许编辑？
- [ ] 编辑中断（未保存）后数据如何处理？
- [ ] 编辑提交后是否影响关联数据？

### 删除 / 注销 / 停用类流程
- [ ] 是否需要二次确认？
- [ ] 被删除的对象有哪些关联影响？是否提示用户？
- [ ] 是否支持恢复？若不支持，是否明确告知用户这是不可逆操作？

## 二、常见信息架构问题

- 同一功能分散在多个入口，但操作逻辑不一致
- 详情页和列表页之间的数据不同步（实时性未定义）
- 权限过滤后，原有页面结构是否仍然合理（避免出现大量空状态或禁用项堆砌）
- 弹窗 / 抽屉的使用场景缺乏统一规则（何时用弹窗，何时用新页面）

---

## 三、高频缺失的边界状态

以下状态在需求文档中最常被遗漏，需逐一核查：

| 状态类型 | 检查点 |
|---|---|
| 空状态 | 列表为空时如何展示？是否有引导？ |
| 加载状态 | 数据加载中的页面表现是否说明？ |
| 错误状态 | 接口失败 / 超时时页面如何处理？ |
| 超限状态 | 数量上限、字符上限、时间范围上限是否说明？ |
| 无权限状态 | 页面不可访问 vs 操作不可用，二者如何区分展示？ |
| 数据异常状态 | 数据格式异常、来源缺失时如何降级展示？ |

---

## 四、设计判断中常见的信息缺口

- 字段是否必填未明确
- 字段最大长度未定义
- 数字字段的单位、精度未说明
- 时间字段的格式、时区未定义
- 图片 / 文件的格式限制、大小限制未说明
- 列表默认排序规则未说明
- 筛选条件的默认值未定义
