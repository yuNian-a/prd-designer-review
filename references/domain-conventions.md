# Domain Conventions

特定行业的常见流程结构与设计惯例，用于在需求预审时判断当前方案是否偏离行业标准模式，以及偏离是否有足够的业务理由。

---

## 使用说明

本文件收录了常见行业场景的设计惯例。AI 在评审时，应对照当前需求所属行业，判断：
- 当前方案是否与行业惯例一致
- 若不一致，是否有明确的业务收益支撑这种差异
- 若没有足够理由，应标记为中或高风险问题

## 一、ToB 管理后台（通用）

### 列表页标准结构
- 顶部：筛选条件区 + 搜索框
- 右上角：主操作按钮（新建 / 导入）
- 表格：数据列表，末列为操作列
- 底部：分页器

### 详情页标准结构
- 顶部：面包屑 + 页面标题 + 操作按钮区
- 主体：分组展示的基本信息
- 关联数据：以 Tab 或分区形式展示

### 审批流标准心智
- 状态通常为：草稿 → 待审批 → 审批中 → 已通过 / 已拒绝
- 拒绝后支持修改重新提交
- 审批记录（时间轴）是高频需求，PRD 经常遗漏

### 权限管理标准结构
- 角色 → 权限集合的映射
- 页面级权限 vs 操作级权限需分开定义
- "可见但不可操作"需要有明确的视觉区分

## 二、支付 / 钱包类产品

### 转账 / 付款流程惯例
1. 输入金额 → 2. 选择收款方 → 3. 确认信息 → 4. 安全验证（密码/生物识别）→ 5. 结果页
- 安全验证步骤不可绕过
- 结果页需展示交易流水号，支持截图分享
- 失败需给出明确原因（余额不足 / 对方账户问题 / 风控拦截等）

### 账户状态惯例
- 正常 / 冻结 / 注销 三态是基本盘
- 冻结状态下，用户能查看但不能操作，需有明确原因提示
- 注销是不可逆操作，通常需要多步确认 + 风险提示

### 余额 / 资产展示惯例
- 支持一键隐藏金额（保护隐私）
- 可用余额 vs 总资产 vs 冻结金额，三个概念需区分展示
- 历史账单需支持按时间范围筛选

## 三、理财 / 基金类产品

### 产品购买流程惯例
1. 产品详情（收益、风险、费率）→ 2. 输入购买金额 → 3. 风险测评确认 → 4. 协议确认 → 5. 支付 → 6. 结果页
- 风险等级匹配不符时，需有明确提示且用户主动确认后才能继续
- 产品协议不可跳过

### 持仓展示惯例
- 总资产 / 今日收益 / 累计收益 是标准三项
- 每只产品的展示至少需要：产品名、现值、收益率、持有份额

## 四、IFA / 顾问服务类产品

### 客户管理标准结构
- 客户列表支持按标签、跟进状态筛选
- 客户详情包含：基本信息 + 资产概览 + 跟进记录 + 持仓
- 跟进记录需支持手动录入，并按时间倒序排列

### 推荐 / 销售流程惯例
- 产品推荐需基于客户风险等级匹配
- 推荐记录需保存（合规要求）
- 客户确认 / 签约流程通常需要留存凭证
